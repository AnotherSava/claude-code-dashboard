**The "dashboard too tall with auto-resize on" bug (2026-07-01).** The window opens at its launch/saved size and never shrinks to fit the rows. Intermittent (~1/3 of launches).

**Root cause:** a startup race. The webview's mount-time `getConfig()` can beat `setup()`'s `app.manage(ConfigState)` (lib.rs), so `get_config` returns `Config::default()` with `auto_resize: None`. The frontend then reads `config.auto_resize === 'none'` and `measureAndSend` early-returns on its very first guard ŌĆö auto-resize is silently disabled for the whole session, so the window is frozen wherever it launched. `show_window` re-emits `config_updated` to fix this, but that event can be missed, AND the **safety-net reveal timer** (lib.rs `spawn`, ~1500ms) revealed the window WITHOUT re-pushing config/setup ŌĆö the gap.

**Fix ŌĆö proper sequencing, no poll (a retry poll was tried first and rejected as a band-aid):**
- Backend `lib.rs`: manage `ConfigState` **first** in `setup()` (moved above the other stores / window setup) so `get_config` is far less likely to hit the `unwrap_or_default()` path. This shrinks the race to near-zero.
- Frontend `App.svelte`: **one authoritative `getConfig()` re-read** at the end of `onMount` (after the other mount round-trips have proven the backend is up), placed before the reveal `finally` so a stalled pre-show `requestAnimationFrame` can't skip it. A single sequenced read, not a loop.
- Backend `lib.rs`: the safety-net reveal timer also calls `emit_config_updated` + `emit_setup_state` (mirroring `show_window`) ŌĆö a cheap backstop.
- Frontend: a `scheduleMeasure()` after `showWindow()` for a reliable post-show measure.
- Verified **9/9 restarts** land `auto_resize:'up'` + a correct apply (was ~1/3 failing).
- Did NOT touch the core `measureAndSend` dedup (a bidirectional-reconcile was tried and reverted ŌĆö the dedup is loop-prone; on a fresh launch `lastSentHeight===-1` so the existing overflow-dedup already shrinks a too-tall window once auto_resize is enabled).
- **Why no clean "backend-then-frontend" barrier:** the webview is coupled to the app lifecycle and can invoke commands (on worker threads) *during* `setup()` ŌĆö there's no natural point to "start the frontend after the backend is ready." The correct pattern is an explicit ready-handshake: init state as early as possible, then have the frontend re-read the authoritative value after a point where the backend has demonstrably answered. (A stricter fully-deterministic option: a `tokio::sync::Notify` set at end of `setup()` + a `wait_until_ready` command the frontend awaits ŌĆö more machinery, not needed here.)

**Follow-up hardening ŌĆö missed *initial* measure now self-heals (2026-07-10).** A deploy relaunch reproduced a stuck-tall window whose signature was DIFFERENT from the config race above: config was `'up'` and the webview cache was freshly wiped (current bundle), yet the post-restart launch logged **zero** `auto_resize measure` and no `mount snapshot` ŌĆö the very first measure early-returned before the widget subtree committed, and nothing re-fired it (every trigger ŌĆö mount, the `$effect`, the ResizeObserver ŌĆö fires once; the collapse heal only rescues too-*short* windows). Fix in `App.svelte`: a bounded **ready-retry** (`READY_RETRY_DELAYS`, ~3.6s budget) that re-attempts `measureAndSend` until the first real measurement lands, then goes quiet (`firstMeasureDone`). `measureAndSend` now splits its early-return into "deliberately off" (`auto_resize==='none'`) vs "not ready yet" (`widgetEl`/`config`/`header`/content-element absent ŌĆö SessionList always renders `.list-inner` or `.empty`, so their absence = uncommitted tree); the not-ready path logs `auto_resize measure not ready` **once** (with `has_widget/has_config/has_header/has_content`) so a future stuck window is self-explaining instead of a silent early-return. **So a stuck-tall window is NO LONGER unrecoverable-without-relaunch** ŌĆö check for the not-ready log to see which piece never bound.

**Debugging gotchas (cost hours):**
- **Multi-window log interleaving:** all four windows (main/about/intensity/history) run `App.svelte` and log to the SAME `widget.jsonl`. A temporary `measure_debug` log looked like the main window failing, but the `config:false`/`mode:none`/`intensityMode:true` lines were the *secondary* windows correctly early-returning. Filter by `!aboutMode && !intensityMode && !historyMode && config` to isolate the main window. **The measure subsystem is now gated to the main window** via `getWindowLabelSync()` (`api.ts`, sync `getCurrentWebviewWindow().label`) ŌĆö `measureAndSend` bails first thing when `!isMainWindow`, so secondary windows no longer emit measure/not-ready lines at all (the ready-retry would otherwise have them log + burn retries forever, since their `.widget` never mounts).
- **Validate via ground truth, not theory:** get the real window height with a Win32 `GetWindowRect` PowerShell probe (Add-Type; run with `-File`, NOT `-ExecutionPolicy Bypass` which the sandbox blocks) and correlate with `auto_resize measure`/`auto_resize::apply` in `widget.jsonl`. "No apply" ŌēĀ "too tall" ŌĆö a window already matching its content correctly skips the apply.
- **A 0-session test artifact:** rapid restarts with no agents reporting can show the SetupPanel (onboarding) via a separate `has_history`/setup_state race, which sizes the window tall to fit the onboarding copy ŌĆö not the session-list too-tall the user reported. Normal use has `sessions>0` so `hookEverReceived` is true and SetupPanel stays hidden. Related: [[debug_state_transitions_via_widget_jsonl]], [[debug_auto_resize_dpi_drift]].
