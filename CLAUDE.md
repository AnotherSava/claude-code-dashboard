# claude-code-dashboard

Tauri v2 desktop dashboard application.

## Project

An always-on-top desktop widget (Windows + macOS) that tracks live status of AI coding agents. A Rust backend (`src-tauri/`) hosts the Tauri v2 app, an embedded `axum` HTTP server on `127.0.0.1:9077` that accepts status POSTs from external agents, a `notify`-backed transcript watcher that tails Claude Code JSONL for live token updates, and a system tray. A Svelte 5 + Vite frontend (`src/`) renders the session list and subscribes to Tauri events for live updates. A thin Python hook (`integrations/claude_hook.py`) forwards Claude Code's raw lifecycle payload to the HTTP API; all classification / chat-id derivation / transcript question-detection lives in `src-tauri/src/adapters/claude.rs`. The `PreToolUse` hook is matcher-gated to `AskUserQuestion`/`ExitPlanMode` only — Claude Code buffers those user-gating tool_use blocks until the user answers, so the JSONL transcript can't carry that signal in flight and `PreToolUse` is the only way to catch them on screen. (`PermissionRequest` and `Elicitation` mark `Awaiting` for the other blocked-on-user cases — tool-permission dialogs and MCP input prompts.) The adapter also handles `UserPromptExpansion` (the early signal a slash command fires before its `!` context-gathering, so a skill launch shows `Working` at once rather than waiting for the lagging `UserPromptSubmit`), `StopFailure` (an API-error turn fires no `Stop`, so this marks the row `Error` with the failure kind), and `PreCompact` (context compaction → a history separator via the new `AdapterOutput::Boundary`). Runtime state lives at `%APPDATA%\com.anothersava.claude-code-dashboard\` (`config.json` for settings, `widget.jsonl` for tracing output).

Key module map in `src-tauri/src/`: `state.rs` (sticky-label state machine + `mark_session_boundary` for mid-session boundary marking), `config.rs` (load/save), `config_watcher.rs` (notify-backed hot-reload), `http_server.rs` (axum handler; on `Clear` it appends `AppState::mark_session_boundary` to the in-memory dialog and persists it before `apply_clear` so the next `SessionStart` restores a dialog ending with the separator and the upcoming `UserPromptSubmit` lands after it — Claude `/clear` fires `SessionEnd`+`SessionStart` so the same cwd-derived chat_id is recreated with the prior dialog restored. Also keeps a defensive `transcript_path`-rotation check before `apply_set` for any case where the path changes without a preceding `SessionEnd`; an `AdapterOutput::Boundary` arm (from `PreCompact`) appends the same separator without ending the session), `log_watcher.rs` (transcript tailing + merge policy + watcher-driven Assistant dialog text upsert, since Claude Code's Stop hook fires before the final assistant turn is flushed to JSONL; for that same too-early-`Stop` reason `Stop` reads the *prior* turn's text and can misclassify the question-vs-done verdict in *either* direction — so once the final turn flushes `apply_and_emit` runs the pure `flushed_turn_verdict` (reusing the adapter's `is_a_question`) and routes to one of two `state` corrections: `promote_done_to_awaiting` (a real question `Stop` settled as `Done` → `Awaiting`) or `demote_scanned_awaiting_to_done` (a statement turn `Stop` stamped `Awaiting "has a question"` after stale-reading a prior question → `Done`). `apply_watcher_update` itself otherwise only ever promotes *to* `Working`, never demotes. That promote-only policy also fixes the background-subagent false-green: when the main turn spawns background agents, its `Stop` fires and settles the row `Done` while the agents keep running — but Claude Code writes a `system`/`turn_duration` record after each turn carrying `pendingBackgroundAgentCount` only while agents are in flight, so `infer_state` reads the newest such record (`InferredState::pending_background`) and forces `Working`, holding the row there (logged as `resume_working` with a background-specific reason) until the final zero-pending turn's `Stop` settles `Done` naturally (Claude-Code-version-specific field — absent → no-op). The demote is gated on `AgentSession::status_from_transcript_scan` — a `#[serde(skip)]` provenance flag the Claude adapter sets only for `Stop`/`idle_prompt` transcript-scan states (via `SetInput::from_transcript_scan`) — so a genuine tool-gated `Awaiting` (`AskUserQuestion`/`PermissionRequest`, which share the `"has a question"` label) is never demoted; both corrections are restricted to their single transition so a transient mid-turn assistant text (row still `Working`) can't trigger either. Also detects the `[Request interrupted by user]` marker an Esc-cancel writes — `infer_state` flags it `ended` and `apply_and_emit` calls `state::revert_cancelled_turn` so the row leaves `Working` without a lifecycle hook, reverting to the status it held *before* the cancelled turn (`AgentSession::status_before_working`, captured by `apply_set` on each non-Working→Working transition) rather than blanket `Idle` — so an aborted reply to a pending question lands back on `Awaiting` and the user's real answer stays an approval cycle instead of clobbering the task; gated by the `detect_cancelled_turns` config flag), `idle_probe.rs` (Windows-only backstop for the *instant* Esc-cancel that writes no transcript marker at all: polls each `Working` session's terminal screen via `terminal_title::read_console_screen` and reverts (via `state::revert_cancelled_turn`, same pre-prompt-status restore as the watcher path) only when it positively sees Claude's idle prompt — the input-box `─` border with no `esc to interrupt`/permission marker — so an unreadable or unrecognized screen can never false-revert. The screen alone isn't enough, though: queuing a *second* prompt mid-turn grows the input box and can scroll the `esc to interrupt` footer past the screen tail, making a busy turn read as idle. Two corroborations guard this. First, a still-queued prompt is detected directly from the transcript: Claude Code writes a `queue-operation` record (`enqueue` when the user queues a prompt mid-turn, `remove` when it is dequeued to run), so an outstanding net-`enqueue` (`has_pending_queued_command`/`queued_pending`, read from the file tail) means the input box is occupied and the demote is suppressed outright while one is live. Second, even with no queued prompt the demote is corroborated against the transcript file's mtime (`current_path` from the `WatcherRegistry`): the pure `idle_step` latches the mtime when the idle streak begins and re-arms instead of demoting if a write lands past that baseline (turn still alive); only an unbroken idle streak over a *quiet*, unqueued transcript demotes, and an unreadable transcript is treated as "can't confirm" and never demotes. Same `detect_cancelled_turns` gate), `tray.rs` (menu + autostart; the "Tray usage badge" submenu writes `config.tray_badge`), `tray_badge.rs` (renders `config.tray_badge` onto the tray icon for a 5h/7d bucket, in two styles. **Lights**: recolor the app icon's three traffic lights by usage — `render_light_badge` detects each lit lamp by saturation + vertical band and repaints it green→amber→red across 6 intervals (`GGG`→…→`YRR`), all-red at 100% (`LIGHT_LEVELS`/`LEVEL_COLORS`; the amber is shifted to gold to read distinctly from red — the icon assets' own middle light is recolored to the same gold). **Number**: draw the zero-padded two-digit percentage rasterized from a bundled font (Bebas Neue, OFL, in `assets/fonts/`) via `fontdue` on a transparent background — only the digits carry alpha, no plate or icon behind them (a solid black plate was tried for contrast, then dropped for the cleaner floating look), sized from text-independent references (tallest glyph + widest 2-char string) so every value shares one height/baseline; at ≥100% it falls back to the all-red light. All drawn at the exact OS tray pixel size (16px×DPI scale on Windows, 18pt×backing scale on macOS — the `tray-icon` crate pins the menu-bar image to 18pt); also sets the tray hover tooltip to both buckets. On top of either style, `refresh` draws a red context-usage alert when any local session reaches `config.tray_context_alert_percent` of its model's window (reusing `notifications::context_percent`; gated on a badge being active — never on `TrayBadge::None` — and a non-zero threshold): the icon-bearing styles (light, the ≥100% fallback, and the plain-icon-no-reading fallback via `render_plain_icon`) draw a red rounded-rect stroke matched to the app icon's outline (`draw_alert_border`, run post-downscale; a cheap parametric two-SDF ring — outer minus inner — sized from the `ICON_HALF_W`/`ICON_HALF_H`/`ICON_RADIUS` fractions measured from the source icon plus a small `ALERT_OUTSET`, with the top/bottom segments 1px thinner than the sides since the icon is portrait), while the number styles frame the digits with a perimeter rounded-rect SDF ring (`overlay_alert_border`). Both reuse the shared `rounded_rect_sdf`. `refresh` is the single entry point, called from `commands::emit_usage_limits_updated`, `commands::emit_sessions_updated` (so token-count changes re-evaluate the context border), the tray submenu, and `config_watcher`. macOS contrast (number mode) and live DPI re-render aren't handled yet — see the `tray_badge_deferred` memory), `commands.rs` (Tauri commands + event emitters), `logging.rs` (tracing setup + `FrontendLogger` that writes IPC log entries directly to `widget.jsonl` matching the tracing JSON envelope — bypasses `tracing::*!` macros because they require static field names). Every status-affecting decision is logged to `widget.jsonl` with a stable `decision` field (greppable) and a human `reason`, keyed by the resolved chat_id: `classify` (adapter event→status; the question path's reason carries the matched `question_reason` rule + an `evidence_snippet` of the assistant text, so "why Awaiting?" is answerable without the transcript — built in `adapters/claude.rs` and emitted from `http_server`'s `event -> set`), `resume_working`/`correct_to_awaiting`/`correct_to_done`/`revert_cancelled` (watcher + `idle_probe` corrections; `revert_cancelled` logs the reverted-to `status`), `apply_set` (the `state.rs` transition), `session_clear`/`compact_boundary`, and `context_alert`/`context_dismiss` (the Telegram context-usage alert send/dismiss lifecycle in `notifications.rs`). The project-local `investigate` skill (`.claude/skills/investigate/`) reconstructs an agent's current state and the decision chain from these lines alone, `notifications.rs` (1s-tick reconciler + `Notifier` trait; per-state firing in the pure `fire_reason` — each `config::StateNotify` carries two independent optional windows and fires when *either* trips: `afk_window_ms` (user idle ≥ window **and** idle ≥ time-in-state, the "no input since the state began = saw it" guard — so a `done` row only pings if you were away when it finished and never came back) or `reaction_window_ms` (state outlasted the backstop regardless of presence, for actionable `awaiting`/`error`). AFK is skipped when idle is unknown; defaults make `done` AFK-only, `awaiting`/`error` AFK + backstop. Also reconciles Telegram context-usage alerts (`context_reconcile`) the same way: sends when a session crosses `context_alert_percent`, tracks the message handle in a `context_outstanding` id→handle map, and deletes the message once the session drops back below / vanishes / the feature is disabled — re-arming after a drop. Both reconcilers revoke through the shared `dismiss_and_forget` helper (delete via `Notifier::dismiss` + drop from the outstanding map; generic over the map's value type via a `handle_of` accessor), differing only in their staleness criteria), `idle.rs` (system-wide input-idle ms via `GetLastInputInfo` on Windows / `CGEventSourceSecondsSinceLastEventType` on macOS, raw FFI like `auto_resize`/`terminal_title`; `None` on query failure or unsupported platform → reconciler treats presence as unknown and skips the AFK path), `prompt_history.rs` (disk persistence for per-session dialog + original_prompt + task_started_at + last_input_tokens — loads/saves `prompt_history.json` in app data dir; `last_input_tokens` is one of two separator triggers — a >50% token drop at app-restart restoration inserts a `DialogRole::Separator` rendered as an `<hr>` in the history window. This branch is dormant today because the Claude adapter doesn't populate hook `input_tokens` — only the watcher does, on `session.input_tokens`. The active path for mid-session boundaries lives in `http_server.rs` above), `chat_id_registry.rs` (persisted `session_id → chat_id` lock in `session_chat_ids.json`; `http_server` calls `resolve` after `dispatch` to override the cwd-derived id so a mid-session `cd` into a subdirectory doesn't fragment one conversation across rows, and `forget` on `SessionEnd` — the cwd-derived id is only the first-seen anchor, which also lets `/clear`'s new session_id re-derive the same name), `custom_names.rs` (`CustomNamesStore` — user-assigned per-`chat_id` display names persisted to `custom_names.json`, resolved onto `AgentSession.display_name` via the shared `CustomNamesStore::apply` — at emit time by `commands::resolved_snapshot` and on the notification path, so a renamed agent reads the same in the dashboard and in Telegram (`AgentSession::display_label` is the name-or-chat_id fallback the notification text builders use); `set_chat_name` command backs the double-click-to-rename in `SessionItem.svelte`. Lives in its own file rather than `config.json` so it survives the deploy step that overwrites config.json), `telegram.rs` (reqwest-based Telegram Bot API client), `terminal_title.rs` (mirrors each session's status onto its terminal tab title as "<colored circle> <name>"; the hook reports candidate pids as `console_pids` in the POST body — console process list + ancestor chain on Windows, ancestor chain alone on macOS — and `sync` — called inside `commands::emit_sessions_updated`, which every state transition already flows through — pushes the title per platform. Windows: the FreeConsole→AttachConsole→SetConsoleTitleW dance, candidates tried far-to-near because hooks are spawned CREATE_NO_WINDOW into a fresh *invisible* console — the transient near-end pids hold that console while the far end fails attach (GUI ancestors) until the first success lands on the user's real terminal (GetConsoleWindow can't discriminate instead: conPTY consoles report no window on current Win11). macOS: resolve a candidate's controlling tty via `ps -o tty=` and write an OSC 0 escape to `/dev/ttysNNN`, near-to-far since dead transients fail the lookup and GUI ancestors report no tty. Successive pid reports are intersected so per-hook transient cmd/python pids drop out and only long-lived ones (Claude itself, the user's shell) survive pid reuse; gated by the `terminal_titles` config toggle, and blank-title cleanup on session disappearance is centralized in `sync` rather than at the Clear call site), `sync.rs` (multi-device session sync — single-writer model: each device pushes full metadata snapshots of its *local* sessions + per-session dialog deltas to `config.sync.peers`, debounced 300ms via the `SyncDirty` Notify poked in `emit_sessions_updated` plus a 30s heartbeat; per-peer watermarks advance only on 2xx so offline windows can't lose entries; deltas ship oldest-first in ~256KB chunks that drain back-to-back once a chunk is acknowledged, so a down peer costs one bounded build per cycle and a returning peer catches up within a cycle; receiver stores into `AppState::remote` — separate from `sessions`, so notifications/persistence/terminal-titles stay remote-blind by construction — with ids namespaced `{device}/{raw_id}` and `origin` stamped, deltas merged via the shared `state::merge_dialog_entries` — but only contiguous ones (the push's `delta_from` watermark must overlap the newest held entry, or be `0` = complete dialog); floating fragments are discarded so held dialogs stay gap-free by construction; bearer-token-gated listener on `0.0.0.0:sync.listen_port` also serves `GET /api/sync/dialog?id=&since=`, the full-dialog catch-up `commands::open_history` triggers for remote targets — bracketed in `history_loading` events the history window renders as a loading hint, and the completeness guarantee for what disk can't cover (fresh install while the origin's watermark is already advanced); 90s TTL reaper drops silent devices; `resolved_snapshot` is the single local+remote merge point; remote-driven mutations — push ingest, TTL reap, dialog catch-up — must emit via `commands::emit_sessions_updated_remote`, which skips the `SyncDirty` poke: a received push poking the pusher echoes the *control signal* even though content can't echo, ping-ponging full snapshots between two devices at the 300ms debounce period), `remote_history.rs` (`RemoteHistoryStore` — disk persistence for remote-device dialogs, one file per device under `remote_history/` in the app data dir; `sync::ingest` re-seeds each session's dialog from it when there's no in-memory predecessor, and `post_sync`/the catch-up save back on new content. Mirrors `prompt_history.rs`, including keep-on-absence), `remote_usage.rs` (`RemoteUsageStore` — disk persistence for remote-device usage-limit samples, one file per device under `remote_usage/`; peers push their `usage_history` records on `SyncPush.usage_delta` (watermark-selected via the pure `sync::usage_delta_since`, carried on the first drain chunk only), `post_sync` merges them in, and `commands::merged_usage_records` unions every device's records into the Work-intensity chart — the account-wide 5h/7d counter means a peer's polls fill the windows this device's app was closed (`get_usage_intensity_week`/`_weeks` build from the union, not just local). A merge re-emits `usage_limits_updated` so an open intensity window re-fetches live. Mirrors `remote_history.rs`), `usage_limits.rs` (Anthropic OAuth usage poller for 5h / 7d rate-limit bars + direct OAuth refresh), `usage_history.rs` (`UsageHistoryStore` — append-only `usage_history.jsonl` in the app data dir, one flat record per successful usage poll with raw API percentages + reset times in epoch ms; the data source for the Work intensity window — the pure `build_week_chart` bins it into a `WeekChart` of 10-min intensity buckets (intensity = clamped Δ of `five_hour_pct` between consecutive polls; resets are detected by a pct drop, never by `five_hour_resets_at`, which jitters ±1min without an actual reset — see the `usage_five_hour_resets_at_jitter` memory)), `auto_resize.rs` (Up/Down content-fit window with edge anchor; Win32 `WM_NCHITTEST`/`WM_NCLBUTTONDOWN` subclass for vertical resize lock; class brush replacement to suppress white flash on horizontal resize — all `#[cfg(windows)]`-gated, no-ops on macOS), `setup.rs` (`include_str!`s `integrations/claude_hook.py` and writes it to `app_data_dir/claude_hook.py` on every launch; builds the `~/.claude/settings.json` snippet for `SetupPanel.svelte`'s copy-to-clipboard button using a `cfg`-gated `python`/`python3` launcher and forward-slash paths so spaces and Windows backslashes survive JSON quoting; `prompt_history::has_any_entries` is the signal that retires `SetupPanel` once the dashboard has ever received a hook event — but `get_setup_state` can be invoked at webview mount before `setup()` manages `PromptHistoryStore`, returning `has_history:false` and flashing the panel on a configured install, so `commands::show_window` re-emits the authoritative `setup_state` event (the frontend registers its listener before calling `show_window`), mirroring the same `emit_config_updated` fix for the `get_config` mount race). On macOS, `lib.rs` setup() calls `app.set_activation_policy(ActivationPolicy::Accessory)` so the app has no Dock icon or menu bar (mirrors the Windows `skipTaskbar` intent). Before the Tauri builder runs, `lib.rs` `clear_webview_cache()` (Windows-only) deletes the WebView2 `EBWebView` cache on every launch: WebView2 caches the fixed-URL `index.html`, so a redeploy/update that only swaps the content-hashed bundle keeps serving stale frontend JS — and the staleness recurs on every launch *within* a single build (so a build-fingerprint gate isn't enough; it must be cleared unconditionally). Must run before window creation because config-defined webviews start loading immediately. The embedded frontend is served from the binary, so the cache buys nothing (see `tauri-webview2-cache-staleness` learning). A second window (label `"history"`, pre-configured hidden in `tauri.conf.json`) shows full dialog history for a session; `HistoryApp.svelte` is its root component. `HistoryTarget` managed state + `history_target` Tauri event pass the session ID from the dashboard click to the history window. `history_font_size` config field (5-step enum) persists font size; tray submenu and Ctrl+/- in the history window both write it. A third window (label `"about"`, decorated + `theme: "Dark"` + `alwaysOnTop` for a float-on-top feel) is the About dialog, surfaced via Help → About; it intentionally does **not** set `parent: "main"` — on macOS NSWindow's `addChildWindow:ordered:` cascades visibility from parent to child, so a parent-child relationship would auto-show About when the main window is first revealed at launch (the App.svelte `aboutMode` gating is sufficient on Windows because Win32 owned windows don't auto-show with their owner, but is insufficient on macOS without removing the parent); `AboutApp.svelte` renders title + version + release date (embedded by `build.rs` via `chrono::Local::now()` in a `cargo:rustc-env`) + clickable docs URL opened in the user's browser via `open::that`. A fourth window (label `"intensity"`, decorated + `theme: "Dark"`) is the Work intensity chart, opened from the tray's **Work intensity** item; `IntensityApp.svelte` is its root — it renders `usage_history` binned by `commands::get_usage_intensity_week`/`get_usage_intensity_weeks` (each → a `WeekChart`; `local_week_start_ms` aligns weeks to local-time Monday 00:00) onto a `<canvas>`, in a **Days** view (7 day-rows, x = time of day) and a scrollable **Weeks** view (one row per week, newest at the bottom, x = Mon→Sun, buckets downsampled ×3); a dashed reference marks the full-5h pace and bars past 2× it clip red. `App.svelte` routes between the four roots via `getWindowLabel()` — the `finally { showWindow() }` block must check **all** non-main mode flags (`!historyMode && !aboutMode && !intensityMode`) or hidden secondaries auto-reveal on every launch. Dynamic `WebviewWindowBuilder` windows were tried first but have broken IPC on Windows — see `tauri-windows-native` learning. All Rust tests live in `#[cfg(test)]` modules next to the code they cover.

## Skills

The project-local `investigate` skill (`.claude/skills/investigate/`) answers "why is agent X in state Y?" from the `decision`-tagged `widget.jsonl` log alone — `investigate.py` resolves the agent (by chat_id, custom display name, or unique substring), reconstructs its current state, and prints the decision chain with reasons; no name lists active sessions to choose from. See the `logging.rs` note in the module map above for the log format it reads.

The `.claude/skills/tauri-*` skills were vendored from https://github.com/dchuk/claude-code-tauri-skills — they are not a verbatim copy. Five skills (`tauri-architecture`, `tauri-capabilities`, `tauri-plugins`, `tauri-project-setup`, `tauri-testing`) were updated with selective improvements from https://github.com/dchuk/claude-code-tauri-skills/pull/2: action-oriented descriptions with explicit trigger keywords, numbered workflows, and build/verify checkpoints. A few reference examples the PR removed were preserved (platform webview table in `tauri-architecture`, extra window/platform examples in `tauri-capabilities`).

Additional hands-on notes from building this project were layered on top of the vendored content:
- `tauri-project-setup` — warns that `create-tauri-app`'s `svelte-ts` template is SvelteKit, not plain Svelte
- `tauri-capabilities` — documents `core:window:allow-start-dragging` for frameless drag regions
- `tauri-system-tray` — adds a `CheckMenuItem` managed-handle pattern for syncing tray checkboxes with their backing state
- `tauri-app-resources` — explains migrating from an existing `.ico` via Pillow and stripping the `android/`/`ios/` output for desktop-only projects
- `tauri-windows-distribution` — documents GitHub Releases' space-to-dot filename normalization for release assets
