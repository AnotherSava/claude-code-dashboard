# claude-code-dashboard

Tauri v2 desktop dashboard application.

## Conventions

**Target only the latest Claude Code release.** The dashboard assumes the current Claude Code hook payloads and JSONL transcript format; we do **not** maintain compatibility shims or fallbacks for older versions. When a newer Claude Code signal supersedes a transcript-scraping or inference workaround, adopt the authoritative signal and delete the workaround outright rather than keeping both (favor clean design over back-compat). For example, the `Stop` hook's `last_assistant_message` and `background_tasks` payload fields replaced the prior too-early-`Stop` transcript-correction machinery and the `pendingBackgroundAgentCount` scrape. (Baseline at adoption: Claude Code 2.1.195.)

## Project

An always-on-top desktop widget (Windows + macOS) that tracks live status of AI coding agents. A Rust backend (`src-tauri/`) hosts the Tauri v2 app, an embedded `axum` HTTP server on `127.0.0.1:9077` that accepts status POSTs from external agents, a `notify`-backed transcript watcher that tails Claude Code JSONL for live token updates, and a system tray. A Svelte 5 + Vite frontend (`src/`) renders the session list and subscribes to Tauri events for live updates. A thin Python hook (`integrations/claude_hook.py`) forwards Claude Code's raw lifecycle payload to the HTTP API; all classification / chat-id derivation / transcript question-detection lives in `src-tauri/src/adapters/claude.rs`. The `PreToolUse` hook is matcher-gated to `AskUserQuestion`/`ExitPlanMode` only — Claude Code buffers those user-gating tool_use blocks until the user answers, so the JSONL transcript can't carry that signal in flight and `PreToolUse` is the only way to catch them on screen. (`PermissionRequest` and `Elicitation` mark `Blocked` for the other blocked-on-user cases — tool-permission dialogs and MCP input prompts; `ElicitationResult` is the authoritative unblock that resumes `Working` the moment the user answers the MCP prompt, rather than waiting for the watcher to infer the resume from the transcript.) The adapter also handles `UserPromptExpansion` (the early signal a slash command fires before its `!` context-gathering, so a skill launch shows `Working` at once rather than waiting for the lagging `UserPromptSubmit`), `StopFailure` (an API-error turn fires no `Stop`, so this marks the row `Error` with the failure kind), and `PreCompact` (context compaction → a history separator via the new `AdapterOutput::Boundary`). Runtime state lives at `%APPDATA%\com.anothersava.claude-code-dashboard\` (`config.json` for settings, `widget.jsonl` for tracing output).

Mirrors `remote_history.rs`), `usage_limits.rs` (Anthropic OAuth usage poller for 5h / 7d rate-limit bars + direct OAuth refresh; the poll loop is reset-aware via the pure `next_poll_secs` — when a bucket's `resets_at` (5h or 7d) falls within one poll interval, it skips the regular poll and instead wakes `config.usage_reset_poll_delay_seconds` (default 30) after the reset, so the bar refreshes off its stale pre-reset value promptly instead of up to a full interval later, and the pre-reset poll is skipped so the two don't fire close together and trip the endpoint's aggressive 429; the delay clears the ±1min `resets_at` jitter + server propagation, and a reset already passed by more than the delay re-polls at the `MIN_POLL_SECS` floor until the fresh window lands), `usage_history.rs` (`UsageHistoryStore` — append-only `usage_history.jsonl` in the app data dir, one flat record per successful usage poll with raw API percentages + reset times in epoch ms; the data source for the Work intensity window — the pure `build_week_chart` bins it into a `WeekChart` of 10-min intensity buckets (intensity = clamped Δ of `five_hour_pct` between consecutive polls; resets are detected by a pct drop, never by `five_hour_resets_at`, which jitters ±1min without an actual reset — see the `usage_five_hour_resets_at_jitter` memory)), `auto_resize.rs` (Up/Down content-fit window with edge anchor; Win32 `WM_NCHITTEST`/`WM_NCLBUTTONDOWN` subclass for vertical resize lock; class brush replacement to suppress white flash on horizontal resize — all `#[cfg(windows)]`-gated, no-ops on macOS), `setup.rs` (`include_str!`s `integrations/claude_hook.py` and writes it to `app_data_dir/claude_hook.py` on every launch; builds the `~/.claude/settings.json` snippet for `SetupPanel.svelte`'s copy-to-clipboard button using a `cfg`-gated `python`/`python3` launcher and forward-slash paths so spaces and Windows backslashes survive JSON quoting; `prompt_history::has_any_entries` is the signal that retires `SetupPanel` once the dashboard has ever received a hook event — and `get_setup_state` returns the correct `has_history` at mount with no re-push, because `PromptHistoryStore` — like every store the frontend reads at mount (`ConfigState`, `FrontendLogger`, the custom-names / usage / remote stores) — is managed in `run()`'s **build()/run() gap**: after `tauri::Builder::build(ctx)` and before `App::run()`, where the `PathResolver` is live (`app.path().app_data_dir()` works) but no webview exists yet (config-defined webviews are created only at the `RunEvent::Ready` that `run()` drives). Managing frontend-read state there is a structural fix for the whole class of mount-time `State` races — `get_config` latching `Config::default()` (auto_resize `None` → frozen window), `frontend_log` dropping every early log, `get_setup_state` flashing the onboarding panel — so `.setup()` (which runs later at Ready, after the webviews exist) does only window/tray/service wiring and reads config from the already-managed state, never managing frontend-read state itself. The former `emit_config_updated` / `emit_setup_state` re-pushes and the frontend authoritative re-read were all removed with this fix. On macOS, `lib.rs` setup() calls `app.set_activation_policy(ActivationPolicy::Accessory)` so the app has no Dock icon or menu bar (mirrors the Windows `skipTaskbar` intent). Before the Tauri builder runs, `lib.rs` `clear_webview_cache()` (Windows-only) deletes the WebView2 `EBWebView` cache on every launch: WebView2 caches the fixed-URL `index.html`, so a redeploy/update that only swaps the content-hashed bundle keeps serving stale frontend JS — and the staleness recurs on every launch *within* a single build (so a build-fingerprint gate isn't enough; it must be cleared unconditionally). Must run before window creation because config-defined webviews start loading immediately. The embedded frontend is served from the binary, so the cache buys nothing (see `tauri-webview2-cache-staleness` learning). A second window (label `"history"`, pre-configured hidden in `tauri.conf.json`) shows full dialog history for a session; `HistoryApp.svelte` is its root component. `HistoryTarget` managed state + `history_target` Tauri event pass the session ID from the dashboard click to the history window. `history_font_size` config field (5-step enum) persists font size; tray submenu and Ctrl+/- in the history window both write it. A third window (label `"about"`, decorated + `theme: "Dark"` + `alwaysOnTop` for a float-on-top feel) is the About dialog, surfaced via Help → About; it intentionally does **not** set `parent: "main"` — on macOS NSWindow's `addChildWindow:ordered:` cascades visibility from parent to child, so a parent-child relationship would auto-show About when the main window is first revealed at launch (the App.svelte `aboutMode` gating is sufficient on Windows because Win32 owned windows don't auto-show with their owner, but is insufficient on macOS without removing the parent); `AboutApp.svelte` renders title + version + release date (embedded by `build.rs` via `chrono::Local::now()` in a `cargo:rustc-env`) + clickable docs URL opened in the user's browser via `open::that`. A fourth window (label `"intensity"`, decorated + `theme: "Dark"`) is the Work intensity chart, opened from the tray's **Work intensity** item; `IntensityApp.svelte` is its root — it renders `usage_history` binned by `commands::get_usage_intensity_week`/`get_usage_intensity_weeks` (each → a `WeekChart`; `local_week_start_ms` aligns weeks to local-time Monday 00:00) onto a `<canvas>`, in a **Days** view (7 day-rows, x = time of day) and a scrollable **Weeks** view (one row per week, newest at the bottom, x = Mon→Sun, buckets downsampled ×3); a dashed reference marks the full-5h pace and bars past 2× it clip red. `App.svelte` routes between the four roots via `getWindowLabel()` — the `finally { showWindow() }` block must check **all** non-main mode flags (`!historyMode && !aboutMode && !intensityMode`) or hidden secondaries auto-reveal on every launch. Dynamic `WebviewWindowBuilder` windows were tried first but have broken IPC on Windows — see `tauri-windows-native` learning. All Rust tests live in `#[cfg(test)]` modules next to the code they cover.

## Skills

The project-local `investigate` skill (`.claude/skills/investigate/`) answers "why is agent X in state Y?" from the `decision`-tagged `widget.jsonl` log alone — `investigate.py` resolves the agent (by chat_id, custom display name, or unique substring), reconstructs its current state, and prints the decision chain with reasons; no name lists active sessions to choose from. See the `logging.rs` note in the module map above for the log format it reads.
