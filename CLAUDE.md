# claude-code-dashboard

Tauri v2 desktop dashboard application.

## Project

An always-on-top desktop widget (Windows + macOS) that tracks live status of AI coding agents. A Rust backend (`src-tauri/`) hosts the Tauri v2 app, an embedded `axum` HTTP server on `127.0.0.1:9077` that accepts status POSTs from external agents, a `notify`-backed transcript watcher that tails Claude Code JSONL for live token updates, and a system tray. A Svelte 5 + Vite frontend (`src/`) renders the session list and subscribes to Tauri events for live updates. A thin Python hook (`integrations/claude_hook.py`) forwards Claude Code's raw lifecycle payload to the HTTP API; all classification / chat-id derivation / transcript question-detection lives in `src-tauri/src/adapters/claude.rs`. The `PreToolUse` hook is matcher-gated to `AskUserQuestion`/`ExitPlanMode` only — Claude Code buffers user-gating tool_use blocks until the user answers, so the JSONL transcript can't carry that signal in flight; this is the only event that can mark the row `Awaiting` while the question is on screen. Runtime state lives at `%APPDATA%\com.anothersava.claude-code-dashboard\` (`config.json` for settings, `widget.jsonl` for tracing output).

Key module map in `src-tauri/src/`: `state.rs` (sticky-label state machine + `mark_session_boundary` for mid-session boundary marking), `config.rs` (load/save), `config_watcher.rs` (notify-backed hot-reload), `http_server.rs` (axum handler; on `Clear` it appends `AppState::mark_session_boundary` to the in-memory dialog and persists it before `apply_clear` so the next `SessionStart` restores a dialog ending with the separator and the upcoming `UserPromptSubmit` lands after it — Claude `/clear` fires `SessionEnd`+`SessionStart` so the same cwd-derived chat_id is recreated with the prior dialog restored. Also keeps a defensive `transcript_path`-rotation check before `apply_set` for any case where the path changes without a preceding `SessionEnd`), `log_watcher.rs` (transcript tailing + merge policy + watcher-driven Assistant dialog text upsert, since Claude Code's Stop hook fires before the final assistant turn is flushed to JSONL), `tray.rs` (menu + autostart), `commands.rs` (Tauri commands + event emitters), `logging.rs` (tracing setup + `FrontendLogger` that writes IPC log entries directly to `widget.jsonl` matching the tracing JSON envelope — bypasses `tracing::*!` macros because they require static field names), `notifications.rs` (1s-tick reconciler + `Notifier` trait), `prompt_history.rs` (disk persistence for per-session dialog + original_prompt + task_started_at + last_input_tokens — loads/saves `prompt_history.json` in app data dir; `last_input_tokens` is one of two separator triggers — a >50% token drop at app-restart restoration inserts a `DialogRole::Separator` rendered as an `<hr>` in the history window. This branch is dormant today because the Claude adapter doesn't populate hook `input_tokens` — only the watcher does, on `session.input_tokens`. The active path for mid-session boundaries lives in `http_server.rs` above), `telegram.rs` (reqwest-based Telegram Bot API client), `usage_limits.rs` (Anthropic OAuth usage poller for 5h / 7d rate-limit bars + direct OAuth refresh), `auto_resize.rs` (Up/Down content-fit window with edge anchor; Win32 `WM_NCHITTEST`/`WM_NCLBUTTONDOWN` subclass for vertical resize lock; class brush replacement to suppress white flash on horizontal resize — all `#[cfg(windows)]`-gated, no-ops on macOS). On macOS, `lib.rs` setup() calls `app.set_activation_policy(ActivationPolicy::Accessory)` so the app has no Dock icon or menu bar (mirrors the Windows `skipTaskbar` intent). A second window (label `"history"`, pre-configured hidden in `tauri.conf.json`) shows full dialog history for a session; `HistoryApp.svelte` is its root component. `HistoryTarget` managed state + `history_target` Tauri event pass the session ID from the dashboard click to the history window. `history_font_size` config field (5-step enum) persists font size; tray submenu and Ctrl+/- in the history window both write it. Dynamic `WebviewWindowBuilder` windows were tried first but have broken IPC on Windows — see `tauri-windows-native` learning. All Rust tests live in `#[cfg(test)]` modules next to the code they cover.

## Skills

The `.claude/skills/tauri-*` skills were vendored from https://github.com/dchuk/claude-code-tauri-skills — they are not a verbatim copy. Five skills (`tauri-architecture`, `tauri-capabilities`, `tauri-plugins`, `tauri-project-setup`, `tauri-testing`) were updated with selective improvements from https://github.com/dchuk/claude-code-tauri-skills/pull/2: action-oriented descriptions with explicit trigger keywords, numbered workflows, and build/verify checkpoints. A few reference examples the PR removed were preserved (platform webview table in `tauri-architecture`, extra window/platform examples in `tauri-capabilities`).

Additional hands-on notes from building this project were layered on top of the vendored content:
- `tauri-project-setup` — warns that `create-tauri-app`'s `svelte-ts` template is SvelteKit, not plain Svelte
- `tauri-capabilities` — documents `core:window:allow-start-dragging` for frameless drag regions
- `tauri-system-tray` — adds a `CheckMenuItem` managed-handle pattern for syncing tray checkboxes with their backing state
- `tauri-app-resources` — explains migrating from an existing `.ico` via Pillow and stripping the `android/`/`ios/` output for desktop-only projects
- `tauri-windows-distribution` — documents GitHub Releases' space-to-dot filename normalization for release assets
