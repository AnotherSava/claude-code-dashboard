# claude-code-dashboard

Tauri v2 desktop dashboard application.

## Conventions

**Target only the latest Claude Code release.** The dashboard assumes the current Claude Code hook payloads and JSONL transcript format; we do **not** maintain compatibility shims or fallbacks for older versions. When a newer Claude Code signal supersedes a transcript-scraping or inference workaround, adopt the authoritative signal and delete the workaround outright rather than keeping both (favor clean design over back-compat). For example, the `Stop` hook's `last_assistant_message` and `background_tasks` payload fields replaced the prior too-early-`Stop` transcript-correction machinery and the `pendingBackgroundAgentCount` scrape. (Baseline at adoption: Claude Code 2.1.195.)

## Project

An always-on-top desktop widget (Windows + macOS) that tracks live status of AI coding agents. A Rust backend (`src-tauri/`) hosts the Tauri v2 app, an embedded `axum` HTTP server on `127.0.0.1:9077` that accepts status POSTs from external agents, a `notify`-backed transcript watcher that tails Claude Code JSONL for live token updates, and a system tray. A Svelte 5 + Vite frontend (`src/`) renders the session list and subscribes to Tauri events for live updates. A thin Python hook (`integrations/claude_hook.py`) forwards Claude Code's raw lifecycle payload to the HTTP API; all classification / chat-id derivation / transcript question-detection lives in `src-tauri/src/adapters/claude.rs`. The `PreToolUse` hook is matcher-gated to `AskUserQuestion`/`ExitPlanMode` only ‚Äî Claude Code buffers those user-gating tool_use blocks until the user answers, so the JSONL transcript can't carry that signal in flight and `PreToolUse` is the only way to catch them on screen. (`PermissionRequest` and `Elicitation` mark `Blocked` for the other blocked-on-user cases ‚Äî tool-permission dialogs and MCP input prompts; `ElicitationResult` is the authoritative unblock that resumes `Working` the moment the user answers the MCP prompt, rather than waiting for the watcher to infer the resume from the transcript.) The adapter also handles `UserPromptExpansion` (the early signal a slash command fires before its `!` context-gathering, so a skill launch shows `Working` at once rather than waiting for the lagging `UserPromptSubmit`), `StopFailure` (an API-error turn fires no `Stop`, so this marks the row `Error` with the failure kind), and `PreCompact` (context compaction ‚Üí a history separator via the new `AdapterOutput::Boundary`). Runtime state lives at `%APPDATA%\com.anothersava.claude-code-dashboard\` (`config.json` for settings, `widget.jsonl` for tracing output).

Key module map in `src-tauri/src/`: `state.rs` (sticky-label state machine + `mark_session_boundary` for mid-session boundary marking), `config.rs` (load/save), `config_watcher.rs` (notify-backed hot-reload), `http_server.rs` (axum handler; on `Clear` it removes the row via the shared `commands::remove_session` ‚Üí `AppState::take_session`, which atomically appends a boundary separator, persists, and drops the row, so the next `SessionStart` restores a dialog ending with the separator and the upcoming `UserPromptSubmit` lands after it ‚Äî Claude `/clear` fires `SessionEnd`+`SessionStart` so the same cwd-derived chat_id is recreated with the prior dialog restored. `remove_session` is shared with the liveness reaper so the SessionEnd path and the exited-without-SessionEnd path can't diverge. An `AdapterOutput::Boundary` arm (from `PreCompact`) appends the same separator without ending the session. `SessionEnd` (clear) and `PreCompact` (compact) are the authoritative boundary signals ‚Äî the prior defensive `transcript_path`-rotation check and the dormant `>50%`-token-drop inference (`state.rs is_new_session`, never fired since the hook path carries no `input_tokens`) were both removed in favor of them), `log_watcher.rs` (transcript tailing + merge policy + watcher-driven Assistant dialog text upsert ‚Äî the dialog text still comes from the watcher because Claude Code's `Stop` hook fires before the final assistant turn is flushed to JSONL, so the post-`Stop` write the `notify` watcher catches is what fills it in. The question-vs-done *verdict* no longer needs the transcript, though: `Stop` carries the final text as `last_assistant_message` ("avoids the need to read and parse the transcript file") and in-flight background work as `background_tasks` (an array, empty/absent when nothing is running), so the adapter's `classify_stop` settles the row authoritatively at `Stop` time ‚Äî `Blocked "has a question"` (a hand-back), else `Waiting` (background work still running), else `Done` ‚Äî and the old too-early-`Stop` correction machinery (a `flushed_turn_verdict` pass plus the `promote_done_to_blocked`/`demote_scanned_blocked_to_done` state corrections and the `status_from_transcript_scan` provenance flag) is gone. `apply_watcher_update` is now purely a promote *to* `Working` (carry a paused row back when the transcript shows the main turn resumed ‚Äî e.g. the user answered an `AskUserQuestion`); it never demotes, and `infer_state` skips `system` and sidechain entries when resolving state, so background-agent (sidechain) activity can't flip a `Waiting` row back to `Working`. The `Waiting` state (the "main turn settled but background subagents are still running" case, rendered light-blue as WAIT) is entered by `classify_stop` from a non-empty `background_tasks` and left when the next turn's `Stop` reports an empty `background_tasks` and settles `Done`. Also detects the `[Request interrupted by user]` marker an Esc-cancel writes ‚Äî `infer_state` flags it `ended` and `apply_and_emit` calls `state::revert_cancelled_turn` so the row leaves `Working` without a lifecycle hook, reverting to the status it held *before* the cancelled turn (`AgentSession::status_before_working`, captured by `apply_set` on each non-Working‚ÜíWorking transition) rather than blanket `Idle` ‚Äî so an aborted reply to a pending question lands back on `Blocked` and the user's real answer stays an approval cycle instead of clobbering the task; gated by the `detect_cancelled_turns` config flag), the former `idle_probe.rs` ‚Äî a Windows-only backstop that read the session's terminal screen and reverted a `Working` row when it positively showed Claude's idle prompt ‚Äî was **removed** in favor of the `log_watcher` marker path above. It rested on the premise that an *instant* Esc-cancel (submit, then cancel before any output) writes nothing to the transcript and fires no hook; real transcripts disprove the first half ‚Äî Claude Code writes `[Request interrupted by user]` even for sub-second, zero-output cancels ‚Äî so the marker path is the **sole, cross-platform Esc-cancel detector** (adopt the authoritative signal, delete the workaround). The screen-scrape was a false-positive-only workaround: a legitimately slow *silent* turn presents the same idle-prompt-over-quiet-transcript picture as an instant cancel, so it could wrongly revert a live turn to `Blocked` ‚Äî a bug no grace window could fix. `liveness_reaper` still covers cancel-then-`exit`, and the next `UserPromptSubmit` re-classifies any unmarked cancel, `liveness.rs` (process-liveness primitives + the `AgentPids` store. The hook now reports `agent_pid` ‚Äî the owning `claude.exe`/`claude` pid, resolved from its ancestor chain by image name ‚Äî on every event; `AgentPids` keeps the latest per chat_id with **overwrite** semantics (not `terminal_title`'s intersection) so a same-cwd restart's new pid supersedes a dead one. `process_images` enumerates running pids‚Üíimage via one Toolhelp32 snapshot on Windows / `ps -axo pid=,comm=` on macOS, and `is_claude_image` confirms an image is the Claude binary ‚Äî so liveness is *image-confirmed*, which defeats both pid reuse (a recycled pid won't read as a live claude) and the `OpenProcess` access-denied trap an elevated claude would hit), `liveness_reaper.rs` (the backstop that removes a row whose owning Claude process exited without a `SessionEnd` ‚Äî which Claude Code fails to deliver on `exit`/Ctrl-D/terminal close, unlike `/clear` (GitHub #69750/#41577), stranding the row; the canonical case is Esc-cancel-then-`exit` (no `SessionEnd`, and the console is gone). A 2s-tick task reaps a local row in **any** state, but only after its `agent_pid` reads dead for `DEAD_STREAK_TO_REAP` consecutive ticks over an *unchanged pid and a quiet `updated`* ‚Äî a still-alive slow session keeps its claude process alive so it can never be reaped, and a same-cwd restart (new live pid / bumped `updated`) restarts the count rather than deleting the fresh row. Removal flows through the shared `commands::remove_session`, guarded by `updated` (via `take_session`) to abort the reap-vs-restart race. ## Skills

The project-local `investigate` skill (`.claude/skills/investigate/`) answers "why is agent X in state Y?" from the `decision`-tagged `widget.jsonl` log alone ‚Äî `investigate.py` resolves the agent (by chat_id, custom display name, or unique substring), reconstructs its current state, and prints the decision chain with reasons; no name lists active sessions to choose from. See the `logging.rs` note in the module map above for the log format it reads.
