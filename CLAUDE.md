# claude-code-dashboard

Tauri v2 desktop dashboard application.

## Project

An always-on-top desktop widget (Windows + macOS) that tracks live status of AI coding agents. A Rust backend (`src-tauri/`) hosts the Tauri v2 app, an embedded `axum` HTTP server on `127.0.0.1:9077` that accepts status POSTs from external agents, a `notify`-backed transcript watcher that tails Claude Code JSONL for live token updates, and a system tray. A Svelte 5 + Vite frontend (`src/`) renders the session list and subscribes to Tauri events for live updates. A thin Python hook (`integrations/claude_hook.py`) forwards Claude Code's raw lifecycle payload to the HTTP API; all classification / chat-id derivation / transcript question-detection lives in `src-tauri/src/adapters/claude.rs`. The `PreToolUse` hook is matcher-gated to `AskUserQuestion`/`ExitPlanMode` only — Claude Code buffers those user-gating tool_use blocks until the user answers, so the JSONL transcript can't carry that signal in flight and `PreToolUse` is the only way to catch them on screen. (`PermissionRequest` and `Elicitation` mark `Awaiting` for the other blocked-on-user cases — tool-permission dialogs and MCP input prompts.) The adapter also handles `UserPromptExpansion` (the early signal a slash command fires before its `!` context-gathering, so a skill launch shows `Working` at once rather than waiting for the lagging `UserPromptSubmit`), `StopFailure` (an API-error turn fires no `Stop`, so this marks the row `Error` with the failure kind), and `PreCompact` (context compaction → a history separator via the new `AdapterOutput::Boundary`). Runtime state lives at `%APPDATA%\com.anothersava.claude-code-dashboard\` (`config.json` for settings, `widget.jsonl` for tracing output).

Key module map in `src-tauri/src/`: `state.rs` (sticky-label state machine + `mark_session_boundary` for mid-session boundary marking), `config.rs` (load/save), `config_watcher.rs` (notify-backed hot-reload), `http_server.rs` (axum handler; on `Clear` it appends `AppState::mark_session_boundary` to the in-memory dialog and persists it before `apply_clear` so the next `SessionStart` restores a dialog ending with the separator and the upcoming `UserPromptSubmit` lands after it — Claude `/clear` fires `SessionEnd`+`SessionStart` so the same cwd-derived chat_id is recreated with the prior dialog restored. Also keeps a defensive `transcript_path`-rotation check before `apply_set` for any case where the path changes without a preceding `SessionEnd`; an `AdapterOutput::Boundary` arm (from `PreCompact`) appends the same separator without ending the session), `log_watcher.rs` (transcript tailing + merge policy + watcher-driven Assistant dialog text upsert, since Claude Code's Stop hook fires before the final assistant turn is flushed to JSONL; for that same too-early-`Stop` reason, once the final turn flushes `apply_and_emit` runs the pure `flushed_turn_is_question` (reusing the adapter's `is_a_question`) and calls `state::promote_done_to_awaiting` to correct a settled `Done` row to `Awaiting` — the one question-ending case `Stop` couldn't see (`apply_watcher_update` otherwise only ever promotes *to* `Working`, never demotes); restricted to the `Done→Awaiting` transition so a transient mid-turn assistant text ending in `?` (row still `Working`) can't false-promote; also detects the `[Request interrupted by user]` marker an Esc-cancel writes — `infer_state` flags it `ended` and `apply_and_emit` calls `state::revert_cancelled_turn` so the row leaves `Working` without a lifecycle hook, reverting to the status it held *before* the cancelled turn (`AgentSession::status_before_working`, captured by `apply_set` on each non-Working→Working transition) rather than blanket `Idle` — so an aborted reply to a pending question lands back on `Awaiting` and the user's real answer stays an approval cycle instead of clobbering the task; gated by the `detect_cancelled_turns` config flag), `idle_probe.rs` (Windows-only backstop for the *instant* Esc-cancel that writes no transcript marker at all: polls each `Working` session's terminal screen via `terminal_title::read_console_screen` and reverts (via `state::revert_cancelled_turn`, same pre-prompt-status restore as the watcher path) only when it positively sees Claude's idle prompt — the input-box `─` border with no `esc to interrupt`/permission marker — so an unreadable or unrecognized screen can never false-revert; same `detect_cancelled_turns` gate), `tray.rs` (menu + autostart; the "Tray usage badge" submenu writes `config.tray_badge`), `tray_badge.rs` (renders `config.tray_badge` onto the tray icon for a 5h/7d bucket, in two styles. **Lights**: recolor the app icon's three traffic lights by usage — `render_light_badge` detects each lit lamp by saturation + vertical band and repaints it green→amber→red across 6 intervals (`GGG`→…→`YRR`), all-red at 100% (`LIGHT_LEVELS`/`LEVEL_COLORS`; the amber is shifted to gold to read distinctly from red — the icon assets' own middle light is recolored to the same gold). **Number**: draw the zero-padded two-digit percentage rasterized from a bundled font (Bebas Neue, OFL, in `assets/fonts/`) via `fontdue` on a transparent background — only the digits carry alpha, no plate or icon behind them (a solid black plate was tried for contrast, then dropped for the cleaner floating look), sized from text-independent references (tallest glyph + widest 2-char string) so every value shares one height/baseline; at ≥100% it falls back to the all-red light. All drawn at the exact OS tray pixel size (16px×DPI scale on Windows, 18pt×backing scale on macOS — the `tray-icon` crate pins the menu-bar image to 18pt); also sets the tray hover tooltip to both buckets. `refresh` is the single entry point, called from `commands::emit_usage_limits_updated`, the tray submenu, and `config_watcher`. macOS contrast (number mode) and live DPI re-render aren't handled yet — see the `tray_badge_deferred` memory), `commands.rs` (Tauri commands + event emitters), `logging.rs` (tracing setup + `FrontendLogger` that writes IPC log entries directly to `widget.jsonl` matching the tracing JSON envelope — bypasses `tracing::*!` macros because they require static field names), `notifications.rs` (1s-tick reconciler + `Notifier` trait; also fires edge-triggered Telegram context-usage alerts when a session crosses `context_alert_percent`), `prompt_history.rs` (disk persistence for per-session dialog + original_prompt + task_started_at + last_input_tokens — loads/saves `prompt_history.json` in app data dir; `last_input_tokens` is one of two separator triggers — a >50% token drop at app-restart restoration inserts a `DialogRole::Separator` rendered as an `<hr>` in the history window. This branch is dormant today because the Claude adapter doesn't populate hook `input_tokens` — only the watcher does, on `session.input_tokens`. The active path for mid-session boundaries lives in `http_server.rs` above), `chat_id_registry.rs` (persisted `session_id → chat_id` lock in `session_chat_ids.json`; `http_server` calls `resolve` after `dispatch` to override the cwd-derived id so a mid-session `cd` into a subdirectory doesn't fragment one conversation across rows, and `forget` on `SessionEnd` — the cwd-derived id is only the first-seen anchor, which also lets `/clear`'s new session_id re-derive the same name), `custom_names.rs` (`CustomNamesStore` — user-assigned per-`chat_id` display names persisted to `custom_names.json`, resolved onto `AgentSession.display_name` at emit time via `commands::resolved_snapshot`; `set_chat_name` command backs the double-click-to-rename in `SessionItem.svelte`. Lives in its own file rather than `config.json` so it survives the deploy step that overwrites config.json), `telegram.rs` (reqwest-based Telegram Bot API client), `terminal_title.rs` (mirrors each session's status onto its terminal tab title as "<colored circle> <name>"; the hook reports candidate pids as `console_pids` in the POST body — console process list + ancestor chain on Windows, ancestor chain alone on macOS — and `sync` — called inside `commands::emit_sessions_updated`, which every state transition already flows through — pushes the title per platform. Windows: the FreeConsole→AttachConsole→SetConsoleTitleW dance, candidates tried far-to-near because hooks are spawned CREATE_NO_WINDOW into a fresh *invisible* console — the transient near-end pids hold that console while the far end fails attach (GUI ancestors) until the first success lands on the user's real terminal (GetConsoleWindow can't discriminate instead: conPTY consoles report no window on current Win11). macOS: resolve a candidate's controlling tty via `ps -o tty=` and write an OSC 0 escape to `/dev/ttysNNN`, near-to-far since dead transients fail the lookup and GUI ancestors report no tty. Successive pid reports are intersected so per-hook transient cmd/python pids drop out and only long-lived ones (Claude itself, the user's shell) survive pid reuse; gated by the `terminal_titles` config toggle, and blank-title cleanup on session disappearance is centralized in `sync` rather than at the Clear call site), `sync.rs` (multi-device session sync — single-writer model: each device pushes full metadata snapshots of its *local* sessions + per-session dialog deltas to `config.sync.peers`, debounced 300ms via the `SyncDirty` Notify poked in `emit_sessions_updated` plus a 30s heartbeat; per-peer watermarks advance only on 2xx so offline windows can't lose entries; deltas ship oldest-first in ~256KB chunks that drain back-to-back once a chunk is acknowledged, so a down peer costs one bounded build per cycle and a returning peer catches up within a cycle; receiver stores into `AppState::remote` — separate from `sessions`, so notifications/persistence/terminal-titles stay remote-blind by construction — with ids namespaced `{device}/{raw_id}` and `origin` stamped, deltas merged via the shared `state::merge_dialog_entries` — but only contiguous ones (the push's `delta_from` watermark must overlap the newest held entry, or be `0` = complete dialog); floating fragments are discarded so held dialogs stay gap-free by construction; bearer-token-gated listener on `0.0.0.0:sync.listen_port` also serves `GET /api/sync/dialog?id=&since=`, the full-dialog catch-up `commands::open_history` triggers for remote targets — bracketed in `history_loading` events the history window renders as a loading hint, and the completeness guarantee for what disk can't cover (fresh install while the origin's watermark is already advanced); 90s TTL reaper drops silent devices; `resolved_snapshot` is the single local+remote merge point; remote-driven mutations — push ingest, TTL reap, dialog catch-up — must emit via `commands::emit_sessions_updated_remote`, which skips the `SyncDirty` poke: a received push poking the pusher echoes the *control signal* even though content can't echo, ping-ponging full snapshots between two devices at the 300ms debounce period), `remote_history.rs` (`RemoteHistoryStore` — disk persistence for remote-device dialogs, one file per device under `remote_history/` in the app data dir; `sync::ingest` re-seeds each session's dialog from it when there's no in-memory predecessor, and `post_sync`/the catch-up save back on new content. Mirrors `prompt_history.rs`, including keep-on-absence), `usage_limits.rs` (Anthropic OAuth usage poller for 5h / 7d rate-limit bars + direct OAuth refresh), `usage_history.rs` (`UsageHistoryStore` — append-only `usage_history.jsonl` in the app data dir, one flat record per successful usage poll with raw API percentages + reset times in epoch ms; data source for future graphs), `auto_resize.rs` (Up/Down content-fit window with edge anchor; Win32 `WM_NCHITTEST`/`WM_NCLBUTTONDOWN` subclass for vertical resize lock; class brush replacement to suppress white flash on horizontal resize — all `#[cfg(windows)]`-gated, no-ops on macOS), `setup.rs` (`include_str!`s `integrations/claude_hook.py` and writes it to `app_data_dir/claude_hook.py` on every launch; builds the `~/.claude/settings.json` snippet for `SetupPanel.svelte`'s copy-to-clipboard button using a `cfg`-gated `python`/`python3` launcher and forward-slash paths so spaces and Windows backslashes survive JSON quoting; `prompt_history::has_any_entries` is the signal that retires `SetupPanel` once the dashboard has ever received a hook event). On macOS, `lib.rs` setup() calls `app.set_activation_policy(ActivationPolicy::Accessory)` so the app has no Dock icon or menu bar (mirrors the Windows `skipTaskbar` intent). Before the Tauri builder runs, `lib.rs` `clear_webview_cache()` (Windows-only) deletes the WebView2 `EBWebView` cache on every launch: WebView2 caches the fixed-URL `index.html`, so a redeploy/update that only swaps the content-hashed bundle keeps serving stale frontend JS — and the staleness recurs on every launch *within* a single build (so a build-fingerprint gate isn't enough; it must be cleared unconditionally). Must run before window creation because config-defined webviews start loading immediately. The embedded frontend is served from the binary, so the cache buys nothing (see `tauri-webview2-cache-staleness` learning). A second window (label `"history"`, pre-configured hidden in `tauri.conf.json`) shows full dialog history for a session; `HistoryApp.svelte` is its root component. `HistoryTarget` managed state + `history_target` Tauri event pass the session ID from the dashboard click to the history window. `history_font_size` config field (5-step enum) persists font size; tray submenu and Ctrl+/- in the history window both write it. A third window (label `"about"`, decorated + `theme: "Dark"` + `alwaysOnTop` for a float-on-top feel) is the About dialog, surfaced via Help → About; it intentionally does **not** set `parent: "main"` — on macOS NSWindow's `addChildWindow:ordered:` cascades visibility from parent to child, so a parent-child relationship would auto-show About when the main window is first revealed at launch (the App.svelte `aboutMode` gating is sufficient on Windows because Win32 owned windows don't auto-show with their owner, but is insufficient on macOS without removing the parent); `AboutApp.svelte` renders title + version + release date (embedded by `build.rs` via `chrono::Local::now()` in a `cargo:rustc-env`) + clickable docs URL opened in the user's browser via `open::that`. `App.svelte` routes between the three roots via `getWindowLabel()` — the `finally { showWindow() }` block must check **all** non-main mode flags (`!historyMode && !aboutMode`) or hidden secondaries auto-reveal on every launch. Dynamic `WebviewWindowBuilder` windows were tried first but have broken IPC on Windows — see `tauri-windows-native` learning. All Rust tests live in `#[cfg(test)]` modules next to the code they cover.

## Skills

The `.claude/skills/tauri-*` skills were vendored from https://github.com/dchuk/claude-code-tauri-skills — they are not a verbatim copy. Five skills (`tauri-architecture`, `tauri-capabilities`, `tauri-plugins`, `tauri-project-setup`, `tauri-testing`) were updated with selective improvements from https://github.com/dchuk/claude-code-tauri-skills/pull/2: action-oriented descriptions with explicit trigger keywords, numbered workflows, and build/verify checkpoints. A few reference examples the PR removed were preserved (platform webview table in `tauri-architecture`, extra window/platform examples in `tauri-capabilities`).

Additional hands-on notes from building this project were layered on top of the vendored content:
- `tauri-project-setup` — warns that `create-tauri-app`'s `svelte-ts` template is SvelteKit, not plain Svelte
- `tauri-capabilities` — documents `core:window:allow-start-dragging` for frameless drag regions
- `tauri-system-tray` — adds a `CheckMenuItem` managed-handle pattern for syncing tray checkboxes with their backing state
- `tauri-app-resources` — explains migrating from an existing `.ico` via Pillow and stripping the `android/`/`ios/` output for desktop-only projects
- `tauri-windows-distribution` — documents GitHub Releases' space-to-dot filename normalization for release assets
